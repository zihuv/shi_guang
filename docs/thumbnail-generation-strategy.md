# 缩略图生成策略设计

## 背景

当前缩略图策略存在几个问题：

- 普通图片缩略图是按需懒生成，但没有明确的“是否值得生成”判定。
- 缩略图尺寸档位较多，实际收益有限，增加了实现复杂度和缓存管理成本。
- 列表与详情侧栏对缩略图的复用边界不够清晰，容易出现小图被大面积放大后发糊。
- 缩略图缓存路径按路径和尺寸分层，目录结构偏复杂。
- 同一图片在不同调用路径下可能出现重复生成。

本次目标是收敛策略，只在确实有性能收益的场景生成缩略图。

## 目标

- 只为特殊文件或超大图片生成缩略图。
- 整个系统只维护一张统一缩略图，列表与详情侧栏共用。
- 缩略图生成逻辑在导入后异步执行，不阻塞主导入流程。
- “是否生成缩略图”的判断抽成独立方法，供导入、重建索引、懒加载等路径复用。
- 尽量避免同一文件重复生成。
- 简化缩略图缓存目录结构。

## 非目标

- 不为所有普通图片预生成缩略图。
- 不为不同视图维护多尺寸缩略图档位。
- 不把缩略图用于原图查看、缩放或沉浸式预览。

## 适用场景

缩略图只用于以下场景：

- 文件列表卡片
- 文件详情侧栏预览

以下场景不使用缩略图：

- 图片查看器中的原图打开
- 放大、缩放、平移
- 导出或分享

## 生成规则

### 一定生成缩略图的类型

以下文件无条件生成缩略图：

- `video`
- `psd`
- `pdf`

规则说明：

- `video`：直接解码首帧或抓帧成本高，缩略图收益明显。
- `psd`：原始文件大，浏览器与主进程直接读取成本高。
- `pdf`：列表和侧栏只需要第一页预览，没有必要每次现场渲染。

### 条件生成缩略图的普通图片

普通位图图片仅在满足任一阈值时生成缩略图：

- 总像素 `>= 10_000_000`，即 `10MP`
- 最长边 `>= 3840`
- 文件体积 `>= 8MB`

建议实现：

```ts
shouldGenerateThumbnail =
  isVideo ||
  ext === "psd" ||
  ext === "pdf" ||
  width * height >= 10_000_000 ||
  Math.max(width, height) >= 3840 ||
  size >= 8 * 1024 * 1024;
```

设计理由：

- 解码成本主要取决于像素数，而不是磁盘压缩后的文件大小。
- 最长边阈值用于覆盖超长图、长截图等极端比例图片。
- 文件体积阈值作为兜底，避免部分压缩不佳的大文件漏判。

### 默认不生成缩略图的情况

以下文件默认不生成缩略图，直接走原文件：

- 常规分辨率的 `jpg/png/webp/avif/gif/svg/bmp/ico/tiff`
- 小尺寸图片
- 小体积图片

## 统一判断方法

需要把“是否生成缩略图”抽成独立可复用方法，放在 Electron 侧公共模块中，由导入、扫描、懒加载和重建索引共同调用。

建议签名：

```ts
export interface ThumbnailDecisionInput {
  ext: string;
  width: number;
  height: number;
  size: number;
}

export interface ThumbnailDecision {
  shouldGenerate: boolean;
  reason:
    | "video"
    | "psd"
    | "pdf"
    | "pixel-threshold"
    | "edge-threshold"
    | "size-threshold"
    | "skip";
}

export function decideThumbnailGeneration(
  input: ThumbnailDecisionInput,
): ThumbnailDecision;
```

设计要求：

- 纯函数，无副作用。
- 不依赖数据库。
- 不依赖调用上下文。
- 可直接单元测试。
- 返回 `reason` 便于日志、调试和统计。

## 缩略图规格

系统只保留一张统一缩略图。

建议规格：

- 格式：`webp`
- 最大边：`768`
- 保持宽高比
- 不放大原图

编码建议：

```ts
.webp({
  quality: 85,
})
```

说明：

- 当前实现使用 `lossless: true`，会明显增加磁盘占用。
- 缩略图的目标是预览，不是保真归档。
- `quality: 85` 对列表和侧栏基本足够，通常能显著降低磁盘占用与读取体积。

如果后续发现细节损失仍可接受，可以在 `82-88` 区间内再做微调。

## 特殊文件的生成方式

### Video

- 使用抓帧生成缩略图。
- 建议抓取较靠前且更稳定的时间点，例如 `0.1s` 或 `duration / 3` 的较小值。

### PDF

- 使用第一页渲染结果生成缩略图。
- 不需要多页缓存，不需要记录页码变体。

### PSD

- 使用合成后的预览图生成缩略图。
- 不做图层级预览。

## 生成时机

### 导入后异步生成

推荐流程：

1. 导入文件
2. 提取基础元数据
3. 生成 `thumbHash`
4. 执行 AI 分析
5. 调用 `decideThumbnailGeneration`
6. 若需要，则将缩略图任务放入后台队列

这样做的原因：

- 用户导入完成得更快
- 缩略图、`thumbHash`、AI 分析都属于后台增强数据，适合统一异步化
- 后续重试机制更清晰

### 懒加载兜底

即使采用导入后异步生成，前端在读取列表卡片或详情侧栏时，仍应保留一次兜底检查：

- 如果数据库标记“应生成缩略图”但缓存文件不存在，则尝试触发一次生成
- 如果数据库标记“不需要缩略图”，则直接读取原图

这样可以覆盖导入中断、任务失败、旧数据升级等场景。

## 去重与避免重复生成

### 缩略图缓存主键

优先使用内容哈希作为缩略图缓存 key：

- `content_hash`

建议缓存文件名：

```txt
.shiguang/thumbs/<content_hash>.webp
```

如果 `content_hash` 暂时不可用，则退回到路径哈希方案，但该方案只作为兼容兜底，不作为长期主路径。

### 进程内并发去重

需要在主进程维护 `in-flight` 任务表，避免同一文件并发生成多次。

建议结构：

```ts
const thumbnailBuildTasks = new Map<string, Promise<string | null>>();
```

key 使用 `content_hash`，若缺失则使用兼容 key。

行为要求：

- 新请求到来时先查 `in-flight`
- 若已有任务，直接复用 Promise
- 任务结束后清理 Map

### 数据库复用

建议在 `files` 表增加或复用字段，明确记录缩略图状态，例如：

- `thumbnail_path`
- `thumbnail_status`
- `thumbnail_updated_at`

如果暂时不扩表，至少应保证可以根据 `content_hash` 快速判断缓存是否已存在。

## 缓存目录结构

目录建议简化为：

```txt
<indexPath>/.shiguang/thumbs/
```

单文件命名：

```txt
<content_hash>.webp
```

不再保留：

- 多尺寸目录
- 基于 `maxEdge` 的变体
- 复杂的嵌套 hash 目录

说明：

- 由于本方案只为部分文件生成缩略图，文件数会显著下降，单目录可接受。
- 如果后续单目录文件量过大，再引入轻量分片，例如按前两位做一级目录。

## 前端使用规则

### 列表卡片

- 若存在缩略图，优先使用缩略图
- 若不存在缩略图，直接使用原图
- 若存在 `thumbHash`，可先显示占位图

### 详情侧栏

- 与列表卡片共用同一张缩略图
- 不额外请求更大版本
- 若缩略图不存在，直接显示原图

### 图片查看器

- 不使用缩略图
- 直接加载原图

## 迁移建议

### 存量数据

需要兼容旧缓存：

- 旧的 `.shiguang/thumbnails/` 可以继续读取，但不再写入
- 新逻辑统一写入 `.shiguang/thumbs/`
- 后续可提供一次清理旧缓存的维护命令

### 旧档位缓存

多尺寸缓存迁移时不需要逐个转换：

- 旧缓存直接废弃
- 让新逻辑按需重新生成单一缩略图即可

## 建议的实现拆分

### 第一阶段

- 新增 `decideThumbnailGeneration`
- 统一缩略图规格为单尺寸
- 普通图片列表与详情侧栏共用同一张缩略图

### 第二阶段

- 把缩略图生成移到导入后异步任务
- 增加主进程 `in-flight` 去重
- 改为基于 `content_hash` 的缓存命名

### 第三阶段

- 支持 `pdf` 第一页缩略图
- 支持 `psd` 合成预览缩略图
- 清理旧缓存目录与旧路径逻辑

## 待确认项

- 统一缩略图最大边是定为 `640` 还是 `768`
- `gif` 是否视为特殊文件单独处理
- `svg` 是否完全跳过缩略图，直接保留原文件渲染
- 是否增加数据库字段记录缩略图状态，还是仅靠缓存文件存在性判断

## 最终结论

建议采用以下收敛方案：

- 只为 `video/psd/pdf` 和超过阈值的大图生成缩略图
- 普通小图直接使用原图
- 系统只保留一张统一缩略图
- 列表与详情侧栏共用同一张缩略图
- 判断逻辑抽成独立方法 `decideThumbnailGeneration`
- 导入后异步生成，并通过 `content_hash` 和 `in-flight` Map 避免重复生成

这套方案能同时解决磁盘占用、生成时机、目录复杂度、重复生成和大图性能问题。
